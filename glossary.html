Kotlin Kotlin Fundamentals 14 terms
val / var
Immutable vs mutable variable declaration
▶
Definition

val declares a read-only variable — its reference cannot be reassigned after initialization. var declares a mutable variable that can be reassigned at any time. In ViewModels, public state is always val (read-only) and private mutable state is also val (the StateFlow itself doesn't change, only its .value does).

Example
val name = "John" // cannot reassign name = "Jane" var count = 0 // can reassign count = 1 private val _query = MutableStateFlow("") // val, but .value can change
📍 Used in every file
by
Property delegate — removes the need to write .value
▶
Definition

by is a Kotlin keyword that delegates a property to another object. In Compose, it is used with collectAsStateWithLifecycle() and remember { mutableStateOf() } so you can write userName directly instead of userName.value everywhere in the composable.

Without vs with by
// Without by — must write .value every time val userName: State<String> = viewModel.userName.collectAsStateWithLifecycle() Text(userName.value) // With by — reads like a plain variable val userName by viewModel.userName.collectAsStateWithLifecycle() Text(userName)
📍 Ch3 ProfileScreen · Ch5 LibraryScreen · Ch7 SearchScreen
data class
Model class with auto-generated equals, hashCode, copy, toString
▶
Definition

A data class is a Kotlin class designed to hold data. The compiler automatically generates equals(), hashCode(), toString(), and most importantly copy(). The copy() function creates a new instance with some fields changed — essential for immutable state updates in the Repository.

Example
data class Playlist( val id: Int, val name: String, val isFavorite: Boolean = false ) // copy() — create a new instance changing only isFavorite val updated = playlist.copy(isFavorite = true)
📍 Ch1 — Playlist.kt · Song.kt
object
Singleton — only one instance exists in the entire app
▶
Definition

object declares a singleton — a class that the Kotlin runtime instantiates exactly once. Every part of the app that references an object gets the same instance. Used for PlaylistRepository so all ViewModels share a single list of playlists.

Example
object PlaylistRepository { // one instance for the entire app private val _playlists = MutableStateFlow(listOf(...)) val playlists = _playlists.asStateFlow() }
📍 Ch1 — PlaylistRepository.kt
?. — Safe call operator
Executes only if the value is not null
▶
Definition

The safe call operator ?. calls a function or accesses a property only if the object is not null. If it is null, the entire expression returns null instead of throwing a NullPointerException.

Example
// currentRoute is null during navigation transitions val currentRoute = currentBackStackEntry?.destination?.route // userName.firstOrNull() returns null if the string is empty userName.firstOrNull()?.uppercaseChar()?.toString() ?: "?"
📍 Ch2 MainScreen · Ch3 ProfileScreen
?: — Elvis operator
Provides a fallback value when the left side is null
▶
Definition

The Elvis operator ?: returns the left side if it is not null, or the right side if it is. It is commonly chained after safe calls to provide a safe default value, avoiding null checks with if/else.

Example
// If getInt() returns null, use 0 as fallback val playlistId = backStackEntry.arguments?.getInt("playlistId") ?: 0 // If the chain returns null, show "?" val initial = userName.firstOrNull()?.uppercaseChar()?.toString() ?: "?"
📍 Ch2 SoundInNavGraph · Ch3 ProfileScreen
?.let { }
Executes a block only when the value is not null
▶
Definition

?.let { } combines a safe call with the let scope function. The block executes only if the value is not null. Inside the lambda, the non-null value is available as it (or a named parameter). Used to conditionally show the PlaylistBottomSheet only when a playlist is selected.

Example
// selectedPlaylist is Playlist? — can be null // The block only runs when the user long-presses a card selectedPlaylist?.let { playlist -> PlaylistBottomSheet( playlist = playlist, onDismiss = { selectedPlaylist = null } ) }
📍 Ch6 LibraryScreen
when
Multi-branch conditional expression — Kotlin's switch
▶
Definition

when is Kotlin's replacement for Java's switch. Unlike switch, when is an expression — it returns a value. Every branch produces a result. Used in LibraryViewModel to decide which list to return based on the selected tab index.

Example
when (tabIndex) { 0 -> playlists // All tab 1 -> playlists.filter { it.isFavorite } // Favorites tab else -> playlists // fallback }
📍 Ch4 LibraryViewModel
filter() · map() · forEach · forEachIndexed
Standard collection operations on lists
▶
Definition

Kotlin's standard library functions for working with collections. All of them receive a lambda and operate on each element.

Each one
// filter — keeps only elements that match the condition playlists.filter { it.isFavorite } // map — transforms each element into something else _playlists.value = _playlists.value.map { if (it.id == playlist.id) it.copy(isFavorite = !it.isFavorite) else it } // forEach — iterates without returning a new list items.forEach { (route, icon, label) -> NavigationBarItem(...) } // forEachIndexed — same as forEach but also gives the index tabs.forEachIndexed { index, title -> Tab(selected = selectedTab == index) }

firstOrNull()
Returns the first element or null if the collection is empty
▶
Definition

Returns the first character (or element) of a string or collection. Unlike first(), it returns null instead of throwing an exception when the string is empty. Safe to chain with ?. operators.

Example
userName.firstOrNull()?.uppercaseChar()?.toString() ?: "?" // "John Doe" → "J" | "" → "?"
📍 Ch3 ProfileScreen — avatar initial
contains() · isBlank() · isEmpty()
String checks used in search filtering
▶
Definition

contains(query, ignoreCase) returns true if the string includes the given substring. isBlank() returns true if the string is empty or contains only whitespace — safer than isEmpty() which only checks for zero characters.

Example
// isBlank() is safer — " " is blank but not empty if (query.isBlank()) { playlists // show all if nothing typed } else { playlists.filter { it.name.contains(query, ignoreCase = true) || it.genre.contains(query, ignoreCase = true) } }
📍 Ch7 SearchViewModel
String templates — ${}
Embed values directly inside strings
▶
Definition

Kotlin string templates allow you to embed expressions directly inside a string using ${expression} or $variable for simple names. The runtime evaluates the expression and converts it to a string automatically.

Example
Text("${playlist.songCount} songs") // "15 songs" navController.navigate("playlistDetail/${playlist.id}") // "playlistDetail/3"

Pair · Triple
Group 2 or 3 values without a custom data class
▶
Definition

Pair and Triple are Kotlin standard library classes that group 2 or 3 values respectively. They support destructuring in lambdas. When you need 4 or more values together, create a data class instead for clarity.

Example

📍 Ch2 BottomNavigationBar
copy()
Create a new data class instance with some fields changed
▶
Definition

Generated automatically by data class. Creates a new instance copying all fields, replacing only the ones you specify. Essential for updating immutable state — you never mutate an existing object, you create a new one with the change applied.

Example
// Toggle isFavorite without mutating the original _playlists.value = _playlists.value.map { if (it.id == playlist.id) it.copy(isFavorite = !it.isFavorite) else it }

Functions & Lambdas Functions & Lambdas 5 terms
() -> Unit
A function type that takes no parameters and returns nothing
▶
Definition

() -> Unit is a Kotlin function type. The parentheses () list the parameter types (empty = no parameters), and Unit is the return type meaning "returns nothing" — equivalent to void in Java. Used for callbacks like onLogout and onDismiss.

Example
fun ProfileScreen( onLogout: () -> Unit // parameter that IS a function ) { Button(onClick = { onLogout() }) { Text("Log Out") } }
📍 Every screen signature in the course
(Playlist) -> Unit · (String) -> Unit
Function types with parameters
▶
Definition

The same pattern as () -> Unit but with typed parameters. (Playlist) -> Unit means "a function that receives a Playlist and returns nothing." These are used for callbacks that need to pass data upward — state hoisting in practice.

Example
fun LibraryScreen( onNavigateToPlaylistDetail: (Playlist) -> Unit // receives a Playlist ) { PlaylistCard( onClick = { onNavigateToPlaylistDetail(playlist) } // passes it up ) }
📍 Ch5 LibraryScreen · Ch7 SearchScreen
Lambda
An anonymous function passed as a value
▶
Definition

A lambda is an anonymous function — a function without a name that you pass as a value. In Kotlin, lambdas are written between curly braces { }. When a lambda is the last parameter of a function, it can be moved outside the parentheses — this is called a trailing lambda.

Regular vs trailing lambda
// Regular lambda as parameter Button(onClick = { viewModel.logout() }) // Trailing lambda — moved outside parentheses Button(onClick = { viewModel.logout() }) { Text("Log Out") // this is also a trailing lambda (content slot) } // The SearchBar trailing lambda holds the results list SearchBar(inputField = { ... }) { LazyColumn { ... } // content shown when SearchBar is expanded }
📍 Every composable in the course
:: — Function reference operator
Pass a function as a value without calling it
▶
Definition

:: creates a reference to a function so it can be passed as a value without being called immediately. When the lambda does nothing except call a single function and pass its argument through directly, :: is a cleaner equivalent.

Lambda vs :: reference
// These two are exactly equivalent: onQueryChange = { query -> viewModel.onQueryChange(query) } onQueryChange = viewModel::onQueryChange // cleaner with :: // Use lambda when the body does something extra onQueryChange = { query -> viewModel.onQueryChange(query) analytics.log("search") // extra work — need full lambda }
📍 Ch7 SearchScreen
Default parameter values — = {}
Make a parameter optional by providing a default
▶
Definition

A parameter with a default value does not need to be passed by the caller. In Compose, function-type parameters often default to {} (an empty lambda that does nothing) so the composable can be used without wiring up every callback — useful for Previews and incremental development.

Example
fun PlaylistCard( playlist: Playlist, onClick: () -> Unit, onLongClick: () -> Unit = {} // optional — defaults to do nothing ) // Can call without onLongClick in Ch4 before BottomSheet exists PlaylistCard(playlist = p, onClick = { ... })

Coroutines & Flow Coroutines & Flow 9 terms
MutableStateFlow
A hot stream whose value can be changed directly
▶
Definition

A MutableStateFlow holds a single value that can be updated via .value =. It is always "hot" — it always has a current value and emits to all active collectors when the value changes. Always declared private in ViewModels so only the ViewModel can modify it.

Example
private val _query = MutableStateFlow("") fun onQueryChange(query: String) { _query.value = query // triggers all collectors }
📍 Ch3 UserSessionViewModel · Ch4 LibraryViewModel · Ch7 SearchViewModel
StateFlow · asStateFlow()
Read-only public view of a MutableStateFlow
▶
Definition

StateFlow is the read-only interface of MutableStateFlow. The UI can observe it but cannot modify it. asStateFlow() converts a MutableStateFlow into a StateFlow — the standard pattern for private mutable state exposed as public read-only state.

Pattern
// private — only ViewModel can write private val _query = MutableStateFlow("") // public — UI can only read val query: StateFlow<String> = _query.asStateFlow()
📍 Every ViewModel in the course
.value
Read or write the current value of a MutableStateFlow
▶
Definition

.value accesses the current value held by a MutableStateFlow. Assigning to it (_query.value = "rock") triggers all collectors — every part of the UI observing this flow updates automatically. Reading from it (_query.value) returns the current value synchronously.

Example
// Write — triggers recomposition in all observers _isLoggedIn.value = true // Read — synchronous, always available val current = _query.value
📍 Every ViewModel in the course
combine()
Merge two or more StateFlows into one reactive result
▶
Definition

combine() takes 2–5 StateFlow sources and a lambda. Every time any source emits a new value, the lambda runs automatically and produces a new result. Think of it as: "whenever X changes OR Y changes, recalculate." The result is a cold Flow — convert it to a hot StateFlow with stateIn().

Example
val filteredPlaylist = combine( PlaylistRepository.playlists, // source 1 _selectedTab // source 2 ) { playlists, tabIndex -> // runs when either changes when (tabIndex) { 0 -> playlists 1 -> playlists.filter { it.isFavorite } else -> playlists } }.stateIn(...)
📍 Ch4 LibraryViewModel · Ch7 SearchViewModel
stateIn()
Convert a cold Flow into a hot StateFlow
▶
Definition

stateIn() converts a cold Flow (like the result of combine()) into a hot StateFlow that always has a value ready for the UI. Takes three parameters: scope, started, and initialValue.

Parameters
.stateIn( scope = viewModelScope, // scope: flow lives as long as the ViewModel started = SharingStarted.WhileSubscribed(5000), // WhileSubscribed(5000): stays active 5s after last // subscriber disconnects — handles screen rotation initialValue = emptyList() // initialValue: what the UI gets before first emission )
📍 Ch4 LibraryViewModel · Ch7 SearchViewModel
Cold Flow vs Hot StateFlow
The fundamental difference between the two stream types
▶
Definition

A cold Flow starts executing only when someone collects it — like a water tap that only flows when opened. Each new collector gets its own independent execution. A hot StateFlow always has a current value regardless of collectors — like a water tank that is always full. stateIn() converts one into the other.

Why it matters
// combine() produces a cold Flow // The UI would only get results while it is collecting // During rotation, the UI briefly disconnects → data lost // stateIn() converts it to a hot StateFlow // Always has the latest value → UI reconnects instantly // WhileSubscribed(5000) keeps it alive during rotation
📍 Ch4 — concept explained in LibraryViewModel
viewModelScope
Coroutine scope tied to the ViewModel lifecycle
▶
Definition

A CoroutineScope provided by the ViewModel class. Coroutines launched in this scope are cancelled automatically when the ViewModel is cleared — preventing memory leaks. Used as the scope parameter of stateIn().

Example
.stateIn( scope = viewModelScope, // flow stops when ViewModel is cleared ... )
📍 Ch4 LibraryViewModel · Ch7 SearchViewModel
suspend function · scope.launch{}
Functions that can pause and resume — run them in a coroutine
▶
Definition

A suspend function can pause execution without blocking the thread and resume when ready. It can only be called from another suspend function or from a coroutine. scope.launch{} starts a coroutine on a given scope. In composables, rememberCoroutineScope() provides a scope tied to the composable's lifecycle.

Example
val scope = rememberCoroutineScope() // sheetState.hide() is suspend — must run in a coroutine modifier = Modifier.clickable { onDelete(playlist) scope.launch { sheetState.hide() } // launches coroutine .invokeOnCompletion { onDismiss() } }
📍 Ch6 PlaylistBottomSheet
invokeOnCompletion{}
Execute a callback when a coroutine Job finishes
▶
Definition

invokeOnCompletion is called on a Job — the object returned by scope.launch{}. The callback runs when the coroutine finishes, whether successfully or cancelled. Used to call onDismiss() only after the sheet close animation has fully completed, preventing a jarring abrupt disappearance.

Why order matters
// Step 1: execute action onToggleFavorite(playlist) // Step 2: animate close (suspend — runs asynchronously) scope.launch { sheetState.hide() } // Step 3: clean up ONLY after animation finishes .invokeOnCompletion { onDismiss() } // If you called onDismiss() directly in step 1, // the sheet disappears instantly with no animation

Compose Compose — State 6 terms
remember { }
Preserve a value across recompositions
▶
Definition

remember caches a value in the composition so it survives recompositions — the repeated calls Compose makes to your function when state changes. Without remember, the value would be reset every time the composable recomposes. It does NOT survive configuration changes like screen rotation — for that, use a ViewModel.

Example
// showLogoutDialog resets to false only when the composable // leaves the composition entirely, not on every recomposition var showLogoutDialog by remember { mutableStateOf(false) } // selectedPlaylist similarly survives within a composition var selectedPlaylist by remember { mutableStateOf<Playlist?>(null) }
📍 Ch3 ProfileScreen · Ch5 LibraryScreen
mutableStateOf()
Local Compose state that triggers recomposition when changed
▶
Definition

Creates a MutableState — a value container that Compose observes. When the value changes, Compose schedules a recomposition of every composable that reads it. Always wrapped in remember { } for local UI state. For state that must survive rotation, use a ViewModel with MutableStateFlow instead.

Example
var showDialog by remember { mutableStateOf(false) } // When this changes, Compose recomposes ProfileScreen Button(onClick = { showDialog = true }) { Text("Log Out") } if (showDialog) { AlertDialog(...) }
📍 Ch3 ProfileScreen · Ch5 LibraryScreen
collectAsStateWithLifecycle()
Collect a StateFlow as Compose State, lifecycle-aware
▶
Definition

Converts a StateFlow from a ViewModel into a Compose State that the UI can observe. Lifecycle-aware — it automatically stops collecting when the screen is not visible (paused or stopped) and resumes when it becomes visible again. This prevents unnecessary background work and potential memory leaks. Preferred over collectAsState().

Example
// StateFlow → Compose State, stops when screen is hidden val userName by sessionViewModel.userName.collectAsStateWithLifecycle() val filteredPlaylist by viewModel.filteredPlaylist.collectAsStateWithLifecycle()
📍 Ch3 ProfileScreen · Ch5 LibraryScreen · Ch7 SearchScreen
Recomposition
Compose re-executes composable functions when their state changes
▶
Definition

When a state value observed by a composable changes, Compose schedules a recomposition — it calls the composable function again with the new state to produce an updated UI. Compose is smart about this: only the composables that actually read the changed value recompose, not the entire screen.

Why key matters
// Without key: Compose may recompose all items on any change items(filteredPlaylist) { playlist -> ... } // With key: only the changed item recomposes items(filteredPlaylist, key = { it.id }) { playlist -> ... }
📍 Concept used in Ch5 LibraryScreen · Ch7 SearchScreen
State hoisting
Move state up so a composable becomes stateless and reusable
▶
Definition

State hoisting is a pattern where state is moved from a composable to its caller. The composable receives the current value and a callback to request changes — it never owns or modifies state directly. This makes composables stateless, reusable, and easier to test. It is the core pattern behind all callback parameters in this course.

Example
// LibraryScreen does not navigate — it reports the tap fun LibraryScreen( onNavigateToPlaylistDetail: (Playlist) -> Unit // hoisted callback ) { PlaylistCard(onClick = { onNavigateToPlaylistDetail(playlist) }) } // MainScreen decides what to do with the tap
📍 Design pattern used in every screen
@OptIn(ExperimentalMaterial3Api::class)
Acknowledge use of an experimental API to suppress compiler warnings
▶
Definition

Some Compose APIs are marked experimental — they are functional but may change in future versions. To use them without a compiler error, you annotate your composable with @OptIn to acknowledge the risk. In this course: ExperimentalMaterial3Api for TopAppBar and SearchBar, ExperimentalFoundationApi for combinedClickable.

Example
@OptIn(ExperimentalMaterial3Api::class) @Composable fun LibraryScreen(...) { ... } // uses TopAppBar @OptIn(ExperimentalFoundationApi::class) @Composable fun PlaylistCard(...) { ... } // uses combinedClickable

Compose Compose — Layout & Modifiers 8 terms
Scaffold · paddingValues
Material 3 layout container that manages TopAppBar, BottomBar, and content insets
▶
Definition

Scaffold is a Material 3 layout that handles the placement of common UI elements — TopAppBar, bottomBar, FloatingActionButton, and snackbarHost. It calculates paddingValues — the insets the content needs to avoid being hidden behind the bars. Always apply paddingValues to the top-level content composable.

Example
Scaffold( topBar = { TopAppBar(...) }, bottomBar = { BottomNavigationBar(...) } ) { paddingValues -> // ALWAYS apply this LazyVerticalGrid( modifier = Modifier.padding(paddingValues) // content starts below TopAppBar ) { ... } }
📍 Ch2 MainScreen · Ch5 LibraryScreen
LazyVerticalGrid · GridCells
Efficient multi-column grid that only renders visible items
▶
Definition

LazyVerticalGrid renders items in a grid layout, only composing the items currently visible on screen. GridCells.Adaptive(minSize) automatically calculates how many columns fit while keeping each cell at least minSize wide — responsive without extra logic. GridCells.Fixed(n) always uses exactly n columns.

Example
LazyVerticalGrid( columns = GridCells.Adaptive(minSize = 160.dp), horizontalArrangement = Arrangement.spacedBy(8.dp), verticalArrangement = Arrangement.spacedBy(8.dp) ) { items(list, key = { it.id }) { item -> PlaylistCard(...) } }
📍 Ch5 LibraryScreen
aspectRatio(1f)
Force height to equal width — makes the composable square
▶
Definition

aspectRatio(ratio) constrains a composable so its height maintains a fixed ratio to its width. A ratio of 1f means height = width — a perfect square. Without it, cards in a grid would have inconsistent heights depending on their content, breaking the visual alignment.

Example
Card( modifier = Modifier .fillMaxWidth() .aspectRatio(1f) // height = width → square card )
📍 Ch4 PlaylistCard
weight(1f)
Fill all available space in a Column or Row
▶
Definition

Inside a Column or Row, weight(n) divides the available space proportionally among items that have a weight. An item with weight(1f) takes all remaining space after items without weight have been measured. Used in PlaylistCard so the color cover box fills the card and the text section only takes what it needs.

Example
Column(modifier = Modifier.fillMaxSize()) { Box(modifier = Modifier.weight(1f)) // takes all remaining space Column(modifier = Modifier.padding(8.dp)) { Text(playlist.name) // only takes what it needs Text("15 songs") } }

fillMaxSize() · fillMaxWidth()
Expand a composable to fill available space
▶
Definition

fillMaxSize() makes a composable fill all available width and height. fillMaxWidth() fills only the width, leaving height determined by content. Used on screen-level composables to fill the display, and on cards to fill their grid cell width.

items(list, key = { it.id })
Iterate a list in a lazy layout with a unique key per item
▶
Definition

TabRow · Tab
Switch between views within the same screen — no NavController
▶
Definition

TabRow renders a horizontal row of tabs with an animated indicator under the selected one. Each Tab inside it shows as selected when its index matches selectedTabIndex. Unlike NavigationBar, TabRow does not involve navigation — it only changes which content is displayed, controlled by state in the ViewModel.

Example

A slide-up modal panel with animated open and close
▶
Definition

Navigation Navigation Compose 8 terms
NavController · NavHostController
The object that manages navigation and back stack
▶
Definition

NavController is the central object for navigation. It tracks the back stack and navigates between composable destinations. Created with rememberNavController() in a composable. NavHostController is the more specific interface used as a parameter type in function signatures. This course uses two independent NavControllers — one in SoundInNavGraph (outer flow) and one in MainScreen (inner flow).

Example
val navController = rememberNavController() navController.navigate(SoundInRoutes.MAIN) navController.popBackStack()

NavHost · composable()
Register all navigation destinations and their screens
▶
Definition

NavHost is the container that maps route strings to composable screens. Each composable(route) inside it registers a destination. When NavController.navigate("library") is called, NavHost finds the matching composable and displays it.

Example
NavHost(navController = navController, startDestination = SoundInRoutes.LOGIN) { composable(SoundInRoutes.LOGIN) { LoginScreen(...) } composable(SoundInRoutes.REGISTER) { RegisterScreen(...) } composable(SoundInRoutes.MAIN) { MainScreen(...) } }

popUpTo() · inclusive
Clear destinations from the back stack when navigating
▶
Definition

popUpTo(route) removes all destinations from the back stack down to the specified route before navigating. inclusive = true also removes that route itself. Used in two situations: preventing return to Login after authentication, and preventing tab stacking in the bottom nav.

Two uses in this course
// Use 1: user cannot go back to Login after authenticating navController.navigate(SoundInRoutes.MAIN) { popUpTo(SoundInRoutes.LOGIN) { inclusive = true } } // Use 2: tabs don't stack — pressing Library 3 times // doesn't create 3 Library screens in the back stack navController.navigate(route) { popUpTo(navController.graph.startDestinationId) { saveState = true } launchSingleTop = true }

Dynamic routes · navArgument() · NavType.IntType
Pass a typed value through the navigation URL
▶
Definition

A dynamic route contains a placeholder in curly braces, e.g. "playlistDetail/{playlistId}". When navigating, the placeholder is replaced with a real value. navArgument() declares the expected argument and its type. NavType.IntType tells Navigation to deserialize the URL string back into an Int.

Example
// Navigate — replace placeholder with real id navController.navigate("playlistDetail/${playlist.id}") // "playlistDetail/3" // Receive — declare and extract the argument composable( route = "playlistDetail/{playlistId}", arguments = listOf(navArgument("playlistId") { type = NavType.IntType }) ) { backStackEntry -> val id = backStackEntry.arguments?.getInt("playlistId") ?: 0 }

Remove the current screen from the back stack and return to the previous one

Definition

Architecture Architecture 5 terms
ViewModel
Survives configuration changes — holds UI state above the composable layer
▶
Definition

A ViewModel is an Android Architecture Component that holds and manages UI-related state. It survives configuration changes like screen rotation — when the screen rotates, composables are destroyed and recreated, but the ViewModel persists. This is why state that must survive rotation (search query, selected tab) lives in a ViewModel, not in a composable with remember.

Example
class LibraryViewModel : ViewModel() { // selectedTab survives rotation — lives in ViewModel private val _selectedTab = MutableStateFlow(0) }
📍 Every ViewModel in the course
Repository Pattern
A single class that owns all data — ViewModels read from it, never manage their own
▶
Definition

The Repository Pattern centralizes data management. Instead of each ViewModel having its own hardcoded list, all ViewModels read from and write to a single Repository. This ensures consistency — deleting a playlist in Library also removes it from Search because both read the same data source.

Without vs with Repository
// Without Repository — two independent lists // Delete in Library → Search still shows deleted playlist ❌ class LibraryViewModel { val playlists = MutableStateFlow(listOf(...)) } class SearchViewModel { val playlists = MutableStateFlow(listOf(...)) } // With Repository — one shared source // Delete in Library → Search updates automatically ✅ class LibraryViewModel { val playlists = PlaylistRepository.playlists } class SearchViewModel { val playlists = PlaylistRepository.playlists }
📍 Ch1 PlaylistRepository
Single source of truth
One authoritative location for each piece of data in the app
▶
Definition

A principle where each piece of data has exactly one owner. Every other part of the app reads from that owner — never keeps a copy. When the owner changes the data, all readers update automatically. In this course: PlaylistRepository is the single source of truth for the playlist list.

Example
// PlaylistRepository owns the list — only it can modify object PlaylistRepository { private val _playlists = MutableStateFlow(listOf(...)) val playlists = _playlists.asStateFlow() // read-only for others fun deletePlaylist(p: Playlist) { ... } // only way to modify }

Activity-level ViewModel
A ViewModel instantiated once and shared across all screens
▶
Definition

When viewModel() is called inside a composable, it associates the ViewModel with the nearest ViewModelStoreOwner. If each screen called viewModel() for UserSessionViewModel independently, they would each get a different instance with different data. By instantiating it once in MainActivity and passing it down as a parameter, all screens share the same instance.

Example
// MainActivity — instantiated once at Activity level val sessionViewModel: UserSessionViewModel = viewModel() // Passed down through SoundInNavGraph → MainScreen → ProfileScreen // All three see the same instance and the same session data

In-memory Repository
A repository that holds data in RAM — the professional pattern before a database exists
▶
Definition

An in-memory repository stores data in RAM using a MutableStateFlow list. Data is lost when the app is closed. Professional Android developers use this pattern when the backend or database is not yet ready. When Room or Retrofit is added later, only the Repository changes — ViewModels and UI remain untouched.

Architecture path
// Current course (UI Advanced) UI → ViewModel → Repository (in-memory MutableStateFlow) // Future courses (Persistence) UI → ViewModel → Repository → Room (local database) // Even later UI → ViewModel → Repository → Retrofit (remote API)

Material 3 Material 3 Color Roles 4 terms
primary · onPrimary
The brand color and what sits on top of it
▶
Definition

primary is the main brand color — used for the TopAppBar background, the TabRow, the avatar circle, and active NavigationBarItem indicators. onPrimary is the color for content placed on top of a primary background — the TopAppBar title, the avatar initial letter. Never use onPrimary on a white background.

Example
TopAppBarDefaults.topAppBarColors( containerColor = MaterialTheme.colorScheme.primary, // purple bar titleContentColor = MaterialTheme.colorScheme.onPrimary // white text )
📍 Ch3 ProfileScreen · Ch5 LibraryScreen · Ch2 BottomNav
onSurface · onSurfaceVariant
Primary and secondary text colors on white backgrounds
▶
Definition

onSurface is the main text color for white or surface-colored backgrounds — high emphasis, for the most important text. onSurfaceVariant is a secondary text color for the same backgrounds — lower emphasis, for supporting or less important information. Using both creates visual hierarchy without changing font size.

Example
// ProfileScreen — name is primary info, email is secondary Text(text = userName, style = MaterialTheme.typography.titleLarge) // uses default onSurface — stands out Text(text = userEmail, color = MaterialTheme.colorScheme.onSurfaceVariant) // explicitly onSurfaceVariant — less prominent // Same pattern in PlaylistCard Text(playlist.name) // onSurface Text("15 songs", color = ...onSurfaceVariant) // secondary

Color roles — the system
How Material 3 color roles create consistent visual hierarchy
▶
Definition

Material 3 defines color roles — semantic names that describe how a color is used, not what color it is. This means the same design works across any theme (light, dark, or custom brand colors) without changing code. Each on* role is always used on top of its corresponding background role.

The rule
// Rule: always pair container + onContainer primary → onPrimary // TopAppBar bg → TopAppBar text surface → onSurface // white bg → main text surface → onSurfaceVariant // white bg → secondary text // Violation example — never do this: Text(color = MaterialTheme.colorScheme.onPrimary) // on white bg → invisible
